📦 FASE 1 — INSTALACIÓN LIMPIA PROFESIONAL

1️⃣ Agregar repositorio oficial PostgreSQL

(Usando Fedora 42 — ajusta si usas otra versión)
sudo dnf install -y https://download.postgresql.org/pub/repos/yum/reporpms/F-42-x86_64/pgdg-fedora-repo-latest.noarch.rpm

2️⃣ Deshabilitar módulo viejo de Fedora

sudo dnf -qy module disable postgresql

3️⃣ Instalar PostgreSQL 17

sudo dnf install -y postgresql17 postgresql17-server postgresql17-contrib

4️⃣ Inicializar clúster de datos

sudo /usr/pgsql-17/bin/postgresql-17-setup initdb

Esto crea:

/var/lib/pgsql/17/data

5️⃣ Arrancar y habilitar servicio

sudo systemctl enable postgresql-17
sudo systemctl start postgresql-17

Verificar:

systemctl status postgresql-17

Debe decir:

Active: active (running)

👤 FASE 2 — USUARIO ADMINISTRADOR DBA

Entrar como postgres (sin password)

sudo -i -u postgres
psql

Crear usuario administrador principal

LOGIN 
PASSWORD 'PostgresStrong#2026'

CREATEDB 
CREATEROLE;

Crear base de datos del administrador

🌐 FASE 3 — CONFIGURACIÓN DE RED

Editar como usuario normal (no postgres)

Editar accesos

(Recomendado luego limitar por IP)

Reiniciar:

🔐 FASE 4 — MODELO DE ROLES PROFESIONAL

Entrar como admin

psql -U ajimenez -h localhost

Crear roles base

CREATE ROLE app_ro NOLOGIN;
CREATE ROLE app_rw NOLOGIN;
CREATE ROLE dba_read NOLOGIN;

Crear usuarios de ejemplo

CREATE ROLE app_user LOGIN PASSWORD 'AppStrong#2026';
GRANT app_rw TO app_user;

CREATE ROLE report_user LOGIN PASSWORD 'Report#2026';
GRANT app_ro TO report_user;

Asignar permisos por esquema

GRANT CONNECT ON DATABASE ajimenez TO app_ro, app_rw, dba_read;

\c ajimenez

GRANT USAGE ON SCHEMA public TO app_ro, app_rw;

GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_ro;
GRANT SELECT,INSERT,UPDATE,DELETE ON ALL TABLES IN SCHEMA public TO app_rw;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
GRANT SELECT ON TABLES TO app_ro;

ALTER DEFAULT PRIVILEGES IN SCHEMA public

💾 FASE 6 — BACKUPS AUTOMÁTICOS PROFESIONALES

Crear carpeta

mkdir -p /home/ajimenez/pg_backups

Script de backup

nano /home/ajimenez/pg_backups/backup_pg.sh

Contenido:

#!/bin/bash

FECHA=$(date +%F_%H-%M)
DEST=/home/ajimenez/pg_backups
DB=ajimenez

pg_dump -U ajimenez $DB | gzip > $DEST/${DB}_$FECHA.sql.gz

find $DEST -type f -mtime +7 -delete

Permisos:

chmod +x /home/ajimenez/pg_backups/backup_pg.sh

Prueba:

/home/ajimenez/pg_backups/backup_pg.sh

Automatizar con cron

crontab -e

Agregar:

0 2 * * * /home/ajimenez/pg_backups/backup_pg.sh

👥 ADMINISTRACIÓN DE USUARIOS, ROLES Y PERMISOS (DBA)

Esta sección es el corazón de la administración profesional en PostgreSQL. Si dominas roles y permisos, dominas la operación diaria de cualquier servidor Postgres en producción.

  • ✅ Separación de responsabilidades (DBA vs App vs Reportes)
  • ✅ Mínimo privilegio (least privilege)
  • ✅ Control de acceso por base de datos, esquema, tabla, secuencia y función
  • ✅ Auditoría de privilegios
  • ✅ Buenas prácticas tipo enterprise

🎯 PRINCIPIOS DE SEGURIDAD (MENTALIDAD DBA)

1) Nunca usar superusuario para aplicaciones

En PostgreSQL, SUPERUSER es equivalente a root. Úsalo solo para tareas administrativas, no para aplicaciones ni reportes.

2) Usar roles “grupo” y usuarios “login”

Patrón recomendado:

  • Roles (NOLOGIN): representan permisos (ej: lectura, escritura, auditoría)
  • Usuarios (LOGIN): cuentas reales que heredan roles

3) Mínimo privilegio

Se otorgan permisos estrictamente necesarios, nada más. En un banco, esta es una regla de vida o muerte.

🧠 CONCEPTOS CLAVE EN POSTGRESQL

✅ Roles y Usuarios

En PostgreSQL, usuarios y roles son lo mismo. La diferencia la define el atributo LOGIN.

  • ROLE ... NOLOGIN → Rol de permisos (grupo)
  • ROLE ... LOGIN → Usuario real (cuenta)

✅ Privilegios por niveles

  • Nivel servidor: superuser, createdb, createrole, replication
  • Nivel base de datos: CONNECT, CREATE, TEMP
  • Nivel esquema: USAGE, CREATE
  • Nivel tablas: SELECT, INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER
  • Nivel secuencias: USAGE, SELECT, UPDATE
  • Nivel funciones: EXECUTE

👑 VALIDACIÓN DEL ADMINISTRADOR (ajimenez)

Conéctate como admin:

psql -U ajimenez -h localhost ajimenez

Ver roles existentes:

\du

Confirmar si ajimenez es superuser:

SELECT rolname, rolsuper, rolcreaterole, rolcreatedb

🏗 MODELO ENTERPRISE RECOMENDADO

Diseño típico de seguridad:

  • DBA Admin: ajimenez (superuser)
  • Aplicación (RW): inserta/actualiza/borra
  • Reportes (RO): solo lectura
  • Monitoreo: lectura a vistas de sistema

Roles (grupos) recomendados:

app_ro      (solo lectura)
app_rw      (lectura/escritura)
dba_read    (monitoreo y vistas)
etl_rw      (procesos ETL)
auditor_ro  (auditoría y compliance)

🧱 CREACIÓN DE ROLES “GRUPO”

Ejecutar como ajimenez:

-- Roles de permisos (NOLOGIN)
CREATE ROLE app_ro NOLOGIN;
CREATE ROLE app_rw NOLOGIN;
CREATE ROLE dba_read NOLOGIN;
CREATE ROLE auditor_ro NOLOGIN;

Nota DBA: estos roles no pueden iniciar sesión. Son “grupos” que acumulan permisos.

👤 CREACIÓN DE USUARIOS “LOGIN”

Crear usuarios reales:

-- Usuario de aplicación
CREATE ROLE app_user LOGIN PASSWORD 'AppStrong#2026';
GRANT app_rw TO app_user;

-- Usuario de reportes
CREATE ROLE report_user LOGIN PASSWORD 'Report#2026';
GRANT app_ro TO report_user;

-- Usuario monitoreo (para exporter, scripts, etc.)
CREATE ROLE monitor_user LOGIN PASSWORD 'Monitor#2026';
GRANT dba_read TO monitor_user;

Buenas prácticas:

  • ✅ Passwords fuertes
  • ✅ Roles por función
  • ✅ No reusar cuentas
  • ✅ En producción: rotación de credenciales

🔐 PERMISOS A NIVEL DE BASE DE DATOS

Otorgar CONNECT a roles:

GRANT CONNECT ON DATABASE ajimenez TO app_ro, app_rw, dba_read, auditor_ro;

Permisos adicionales (si aplica):

-- Permitir CREATE (solo si realmente lo necesitas)
-- GRANT CREATE ON DATABASE ajimenez TO app_rw;

-- Permitir TEMP (raramente necesario)
-- GRANT TEMP ON DATABASE ajimenez TO app_rw;

🗂 PERMISOS A NIVEL DE ESQUEMA

PostgreSQL organiza objetos dentro de esquemas. El esquema por defecto es public. En entornos enterprise se recomienda crear un esquema dedicado para la app.

✅ Crear esquema dedicado

CREATE SCHEMA app AUTHORIZATION ajimenez;

✅ Revocar permisos innecesarios en public

REVOKE CREATE ON SCHEMA public FROM PUBLIC;
REVOKE ALL ON SCHEMA public FROM PUBLIC;

✅ Otorgar acceso al esquema app

GRANT USAGE ON SCHEMA app TO app_ro, app_rw, dba_read, auditor_ro;
GRANT CREATE ON SCHEMA app TO app_rw; -- solo si la app crea tablas/objetos

Nota DBA: USAGE permite “ver” objetos en el esquema. CREATE permite crear objetos dentro del esquema.

📄 PERMISOS A NIVEL DE TABLAS

✅ Permisos actuales (tablas ya existentes)

-- Solo lectura
GRANT SELECT ON ALL TABLES IN SCHEMA app TO app_ro;

-- Lectura/escritura
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO app_rw;

✅ Permisos futuros (tablas que se creen después)

Esto es clave para administración real: sin “default privileges”, cada tabla nueva nacerá sin permisos y la app fallará. 

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT SELECT ON TABLES TO app_ro;

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_rw;

🔢 PERMISOS A NIVEL DE SECUENCIAS

Muchas tablas usan SERIAL o IDENTITY, que dependen de secuencias. Si no otorgas permisos, la app puede fallar al insertar.

✅ Permisos actuales (secuencias existentes)

GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA app TO app_rw;
GRANT SELECT ON ALL SEQUENCES IN SCHEMA app TO app_ro;

✅ Permisos futuros (secuencias nuevas)

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT USAGE, SELECT ON SEQUENCES TO app_rw;

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT SELECT ON SEQUENCES TO app_ro;

🧩 PERMISOS A NIVEL DE FUNCIONES Y PROCEDIMIENTOS

En PostgreSQL, las funciones pueden ejecutar lógica crítica. Recomendación: conceder EXECUTE solo a quien lo necesita.

Permitir ejecución (actual y futuro)

GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA app TO app_rw;
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA app TO app_ro;

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT EXECUTE ON FUNCTIONS TO app_rw;

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT EXECUTE ON FUNCTIONS TO app_ro;

🔎 AUDITORÍA Y VISIBILIDAD DE PERMISOS

Listar permisos por tabla (útil en auditoría)

SELECT grantee, table_schema, table_name, privilege_type
FROM information_schema.table_privileges
WHERE table_schema = 'app'
ORDER BY table_name, grantee, privilege_type;

Ver membresía de roles

SELECT
  r.rolname as role,
  m.rolname as member
FROM pg_auth_members am
JOIN pg_roles r ON am.roleid = r.oid
JOIN pg_roles m ON am.member = m.oid
ORDER BY r.rolname, m.rolname;

🚧 LIMITES DE CONEXIÓN Y CONTROL DE RECURSOS

Limitar conexiones por usuario:

ALTER ROLE app_user CONNECTION LIMIT 20;
ALTER ROLE report_user CONNECTION LIMIT 5;
ALTER ROLE monitor_user CONNECTION LIMIT 3;

Nota DBA: esto evita que una app con bug consuma todas las conexiones del servidor.

🛑 ERRORES COMUNES Y CÓMO RESOLVERLOS

1) “permission denied for schema …”

Falta USAGE en el esquema:

GRANT USAGE ON SCHEMA app TO app_rw;

2) “permission denied for relation …”

Falta permiso en tablas:

GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA app TO app_rw;

3) Inserts fallan por secuencias

Falta permiso en secuencias:

GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA app TO app_rw;

4) Tablas nuevas no heredan permisos

Falta ALTER DEFAULT PRIVILEGES:

ALTER DEFAULT PRIVILEGES IN SCHEMA app
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_rw;

🧠 CONFIGURACIÓN DE MEMORIA (CORE PERFORMANCE)

shared_buffers

Memoria dedicada a caché interna de Postgres.

shared_buffers = 25%RAM

Ejemplo 16GB RAM → 4GB

effective_cache_size

Estimación de caché del sistema operativo + Postgres.

effective_cache_size = 75%RAM

work_mem

Memoria por operación (sort, hash, joins).

work_mem = 64MB

⚠ Se multiplica por número de operaciones concurrentes.

maintenance_work_mem

Memoria para VACUUM, CREATE INDEX, ANALYZE.

maintenance_work_mem = 512MB

💾 WAL Y CHECKPOINTS

wal_buffers = 16MB
checkpoint_completion_target = 0.9
max_wal_size = 4GB
min_wal_size = 1GB

Esto reduce I/O agresivo y evita picos de disco.

📈 MONITOREO DE PERFORMANCE NATIVO

Sesiones activas

SELECT pid, usename, state, wait_event, query
FROM pg_stat_activity;

Queries más costosas

SELECT query, calls, total_time, mean_time
FROM pg_stat_statements
ORDER BY total_time DESC
LIMIT 10;

Índice cache hit ratio

SELECT
  sum(idx_blks_hit) * 100.0 / nullif(sum(idx_blks_hit + idx_blks_read),0)
  AS index_hit_ratio
FROM pg_statio_user_indexes;

Ideal > 99%

🗂 DISEÑO DE ÍNDICES (CRÍTICO)

Índice básico

CREATE INDEX idx_clientes_email ON app.clientes(email);

Índice compuesto

CREATE INDEX idx_orders_cliente_fecha 
ON app.orders(cliente_id, fecha);

Índice parcial

CREATE INDEX idx_orders_activos 
ON app.orders(fecha)
WHERE estado = 'ACTIVO';

Índice con expresión

📄 BACKUP LÓGICO PROFESIONAL (pg_dump)

Backup completo de una base

pg_dump -U ajimenez ajimenez > ajimenez_full.sql

Con compresión

pg_dump -U ajimenez ajimenez | gzip > ajimenez_full.sql.gz

Solo estructura

pg_dump -U ajimenez -s ajimenez > schema.sql

Solo datos

pg_dump -U ajimenez -a ajimenez > data.sql

Solo una tabla

pg_dump -U ajimenez -t app.clientes ajimenez > clientes.sql

♻ RESTAURACIÓN LÓGICA

Restaurar completa

psql -U ajimenez ajimenez < ajimenez_full.sql

Restaurar comprimido

gunzip -c ajimenez_full.sql.gz | psql -U ajimenez ajimenez

Restaurar tabla específica

psql -U ajimenez ajimenez < clientes.sql

🌍 BACKUP GLOBAL (ROLES + PERMISOS)

PostgreSQL no guarda usuarios dentro de cada base. Por eso se debe respaldar todo el cluster: 

pg_dumpall -U ajimenez > cluster_full.sql

Restaurar:

psql -U ajimenez postgres < cluster_full.sql

⏪ POINT IN TIME RECOVERY (PITR) EN POSTGRESQL — RECUPERACIÓN AVANZADA DBA

PITR permite restaurar PostgreSQL exactamente a un momento específico del pasado. Es más potente que cualquier backup tradicional y es obligatorio en entornos empresariales.

  • ✅ Recuperar borrados accidentales
  • ✅ Volver segundos antes de un error
  • ✅ Recuperación ante corrupción
  • ✅ Continuidad del negocio

🧠 CONCEPTO CLAVE DE PITR

PostgreSQL usa WAL (Write Ahead Log) para registrar cada cambio antes de escribir en disco. PITR combina:

  • Backup base completo
  • Archivos WAL archivados continuamente

Con esto se puede reproducir la historia de la base de datos segundo por segundo.

📦 COMPONENTES NECESARIOS

  • 📁 Backup base (pg_basebackup)
  • 📁 Archivo de WAL históricos
  • ⚙️ Configuración de archivado
  • 🧭 Punto objetivo de recuperación

⚙️ PASO 1 — HABILITAR ARCHIVADO WAL

sudo nano /var/lib/pgsql/17/data/postgresql.conf

Agregar o modificar:

wal_level = replica
archive_mode = on
archive_command = 'cp %p /home/ajimenez/pg_wal_archive/%f'
archive_timeout = 60

Crear directorio WAL:

mkdir -p /home/ajimenez/pg_wal_archive
chown -R ajimenez:ajimenez /home/ajimenez/pg_wal_archive

Reiniciar:

sudo systemctl restart postgresql-17

🧪 PASO 2 — GENERAR ACTIVIDAD DE PRUEBA

psql -U ajimenez -h localhost ajimenez
CREATE TABLE app.pitr_test(id int, descripcion text);
INSERT INTO app.pitr_test VALUES (1,'ok'),(2,'ok'),(3,'ok');

📸 PASO 3 — BACKUP BASE COMPLETO

pg_basebackup -U ajimenez -D /home/ajimenez/base_backup -Fp -Xs -P

Este es el punto inicial de recuperación.

💣 PASO 4 — SIMULAR DESASTRE

DROP TABLE app.pitr_test;

Confirmar:

\dt app.*

⏪ PASO 5 — PROCESO DE RECUPERACIÓN PITR

1️⃣ Detener PostgreSQL

sudo systemctl stop postgresql-17

2️⃣ Limpiar data actual

sudo rm -rf /var/lib/pgsql/17/data/*

3️⃣ Restaurar backup base

sudo cp -r /home/ajimenez/base_backup/* /var/lib/pgsql/17/data/
sudo chown -R postgres:postgres /var/lib/pgsql/17/data

4️⃣ Configurar recuperación

sudo nano /var/lib/pgsql/17/data/postgresql.conf
restore_command = 'cp /home/ajimenez/pg_wal_archive/%f %p'
recovery_target = 'immediate'

5️⃣ Crear señal de recuperación

sudo touch /var/lib/pgsql/17/data/recovery.signal

6️⃣ Arrancar PostgreSQL

sudo systemctl start postgresql-17

✅ PASO 6 — VALIDAR RECUPERACIÓN

psql -U ajimenez -h localhost ajimenez
SELECT * FROM app.pitr_test;

🎉 La tabla debe existir nuevamente.

🎯 RECUPERAR A UNA HORA EXACTA

recovery_target_time = '2026-01-25 14:05:00'

PostgreSQL se detendrá justo en ese segundo.

📊 MONITOREO DE WAL

SELECT * FROM pg_stat_archiver;

Debe mostrar WAL archivándose correctamente.

⚠ ERRORES COMUNES PITR

  • ❌ No probar restauración
  • ❌ No respaldar WAL
  • ❌ Borrar WAL antiguos
  • ❌ No tener backup base
  • ❌ Permisos incorrectos

  • ✅ WAL archivados
  • ✅ Backup base regular
  • ✅ Restauración probada
  • ✅ Retención definida

🔄 REPLICACIÓN STREAMING EN POSTGRESQL (MASTER → STANDBY) — MANUAL DBA PASO A PASO

Este capítulo es una guía completa, práctica y de principio a fin para implementar replicación streaming en PostgreSQL 17 como en un entorno real de producción.

  • ✅ Alta disponibilidad (HA) básica
  • ✅ Réplica en tiempo real
  • ✅ Base para failover automático
  • ✅ Lecturas en standby
  • ✅ Preparación para DR (disaster recovery)

🏗 ARQUITECTURA Y CONCEPTOS

¿Qué es streaming replication?

PostgreSQL replica cambios del servidor primario (PRIMARY / MASTER) hacia un servidor secundario (STANDBY) enviando continuamente los registros WAL. 

PRIMARY  ──(WAL Streaming)──▶  STANDBY
  RW                          RO (opcional)

Objetivo

  • ✔ Si el primary falla, el standby puede promoverse manualmente o por herramientas (Patroni, etc.)
  • ✔ Se reduce riesgo de pérdida de datos

✅ REQUISITOS Y PREPARACIÓN

Requisitos mínimos

  • ✔ Dos servidores (recomendado) o dos instancias (para laboratorio)
  • ✔ PostgreSQL 17 instalado en ambos
  • ✔ Conectividad TCP entre nodos (puerto 5432)
  • ✔ Ajustes de red y firewall
  • ✔ Usuario de replicación

Nomenclatura (ejemplo)

  • PRIMARY (Servidor 1): 192.168.1.10
  • STANDBY (Servidor 2): 192.168.1.11
  • Puerto: 5432
  • Usuario DBA: ajimenez
  • Usuario réplica: replica_user
⚠ Ajusta IPs según tu red. Si estás haciendo laboratorio en una sola máquina, necesitas puertos distintos y data_dir distintos. En producción real, se usan servidores separados.

🧱 PARTE A — CONFIGURAR EL PRIMARY (MASTER)

Paso A1 — Confirmar que el servicio está activo

systemctl status postgresql-17

Debe ser:

Active: active (running)

Paso A2 — Configuración de red del primary

Editar:

sudo nano /var/lib/pgsql/17/data/postgresql.conf

Confirmar:

listen_addresses = '*'
port = 5432

Paso A3 — Activar parámetros de replicación en primary

En el mismo archivo postgresql.conf, agregar o confirmar:

# --- Replicación ---
wal_level = replica
max_wal_senders = 10
max_replication_slots = 10
hot_standby = on

# Mantener WAL suficientes para evitar pérdida si standby se atrasa
wal_keep_size = 1024MB

# Opcional: mejora de checkpoints (reduce spikes)
checkpoint_completion_target = 0.9
max_wal_size = 4GB
min_wal_size = 1GB

Explicación DBA:
- wal_level=replica activa WAL suficiente para replicación.
- max_wal_senders define cuántos procesos pueden enviar WAL.
- wal_keep_size evita que el primary borre WAL antes que standby los consuma (aunque en producción se recomienda slot).
- max_replication_slots habilita slots (recomendado).

Paso A4 — Crear usuario de replicación

psql -U ajimenez -h localhost postgres
CREATE ROLE replica_user
LOGIN
REPLICATION
PASSWORD 'ReplicaStrong#2026';

Nota DBA: Este usuario solo debe tener permisos de replicación, no acceso a tablas.

Paso A5 — Configurar pg_hba.conf para permitir replicación

sudo nano /var/lib/pgsql/17/data/pg_hba.conf

Agregar una regla para el standby (recomendado por IP exacta):

# Permitir replicación desde el standby
host    replication     replica_user     192.168.1.11/32     scram-sha-256

Si todavía no usas SCRAM, puedes poner md5, pero lo recomendado es SCRAM.

Paso A6 — Reiniciar el primary

sudo systemctl restart postgresql-17

Paso A7 — Verificar que parámetros aplicaron

psql -U ajimenez -h localhost postgres
SHOW wal_level;
SHOW max_wal_senders;
SHOW max_replication_slots;
SHOW hot_standby;
SHOW wal_keep_size;

Paso A8 — Firewall (si aplica)

Permitir acceso del standby al puerto 5432 (ejemplo con firewalld):

sudo firewall-cmd --permanent --add-rich-rule='
rule family="ipv4"
source address="192.168.1.11/32"
port protocol="tcp" port="5432" accept'
sudo firewall-cmd --reload

🧩 PARTE B — PREPARAR EL STANDBY (REPLICA)

Paso B1 — Instalar PostgreSQL 17 (si aún no está)

sudo dnf install -y postgresql17 postgresql17-server postgresql17-contrib

Paso B2 — Detener servicio en standby

sudo systemctl stop postgresql-17

Paso B3 — Limpiar data_dir del standby

sudo rm -rf /var/lib/pgsql/17/data/*
⚠ Este paso elimina cualquier contenido. En producción el standby debe estar vacío antes de clonar.

Paso B4 — Clonar el primary con pg_basebackup

Desde el standby, ejecutar: 

pg_basebackup \
-h 192.168.1.10 \
-p 5432 \
-U replica_user \
-D /var/lib/pgsql/17/data \
-Fp -Xs -P -R

Explicación DBA:
- -D destino del data_dir
- -Xs incluye WAL necesarios para consistencia
- -P muestra progreso
- -R crea automáticamente standby.signal y la configuración de conexión al primary

Paso B5 — Verificar archivos creados

ls -lh /var/lib/pgsql/17/data | egrep "standby.signal|postgresql.auto.conf"

Debe existir:

  • standby.signal → indica modo réplica
  • postgresql.auto.conf → contiene primary_conninfo

Paso B6 — Asegurar permisos correctos

sudo chown -R postgres:postgres /var/lib/pgsql/17/data
sudo chmod 700 /var/lib/pgsql/17/data

Paso B7 — Arrancar el standby

sudo systemctl start postgresql-17

Paso B8 — Verificar que el standby está en modo recuperación

psql -U ajimenez -h localhost postgres
SELECT pg_is_in_recovery();

Debe devolver:

true

🔎 PARTE C — VALIDACIONES COMPLETAS (OBLIGATORIO EN PRODUCCIÓN)

Paso C1 — Ver conexión en el primary

psql -U ajimenez -h localhost postgres
SELECT client_addr, state, sync_state, write_lag, flush_lag, replay_lag
FROM pg_stat_replication;

Debes ver el standby conectado. El estado típico: streaming

Paso C2 — Ver estado en el standby

SELECT status, receive_start_lsn, received_lsn, latest_end_lsn
FROM pg_stat_wal_receiver;

Debe mostrar:

  • status = streaming
  • LSN avanzando

Paso C3 — Prueba real de replicación (datos)

En el primary:

psql -U ajimenez -h localhost ajimenez

  id serial primary key,

🧠 PARTE D — CONFIGURAR SÍNCRONA VS ASÍNCRONA

Asíncrona (default)

El primary confirma commits sin esperar al standby. Ventaja: máximo rendimiento.
Desventaja: riesgo mínimo de pérdida si el primary cae justo al escribir.

Síncrona (enterprise)

El primary espera confirmación del standby antes de confirmar commit. Ventaja: cero pérdida de datos (si la red está bien).
Desventaja: más latencia.

Habilitar síncrona en primary

sudo nano /var/lib/pgsql/17/data/postgresql.conf
synchronous_commit = on
synchronous_standby_names = '1 (*)'

Reiniciar:

sudo systemctl restart postgresql-17

🧷 PARTE E — RECOMENDACIÓN ENTERPRISE: REPLICATION SLOTS

Los slots evitan pérdida de WAL si el standby se desconecta. Sin slot, si el standby se atrasa demasiado, el primary puede reciclar WAL y romper la réplica.

Crear slot en primary

SELECT * FROM pg_create_physical_replication_slot('standby01');

Usar slot en standby

Editar en standby el postgresql.auto.conf (si no existe, crear)

sudo nano /var/lib/pgsql/17/data/postgresql.auto.conf
primary_slot_name = 'standby01'

Reiniciar standby:

sudo systemctl restart postgresql-17

Ver slots en primary

SELECT slot_name, active, restart_lsn
FROM pg_replication_slots;

  • Confirmar método (scram/md5)
  • Revisar logs

2) “FATAL: could not connect” (red/firewall)

3) Replica se cae por WAL faltantes

  • Aumentar wal_keep_size

Auditoría de Logins Fallidos en PostgreSQL 17

Implementación para registrar automáticamente en una tabla los intentos fallidos de autenticación al servidor PostgreSQL, leyendo eventos desde los logs del sistema e insertándolos en auditoria.failed_logins.

1) Objetivo

  • Registrar fecha y hora del evento (event_time)
  • Usuario (username)
  • Dirección IP (client_ip)
  • Mensaje de error (error_message)
  • Línea original del log (log_line)
  • Fecha de inserción (inserted_at)

2) Creación del esquema y tabla

2.1 Crear esquema y tabla

CREATE SCHEMA IF NOT EXISTS auditoria;

CREATE TABLE auditoria.failed_logins (
	id bigserial NOT NULL,
	event_time timestamp NULL,
	username text NULL,
	database_name text NULL,
	client_ip text NULL,
	error_message text NULL,
	log_line text NULL,
	inserted_at timestamp DEFAULT CURRENT_TIMESTAMP NULL,
	CONSTRAINT failed_logins_pkey PRIMARY KEY (id)
);

2.2 Constraint UNIQUE (requerido para ON CONFLICT)

ALTER TABLE auditoria.failed_logins
ADD CONSTRAINT failed_logins_log_line_unique UNIQUE (log_line);

3) Crear usuario auditoria con password del script

CREATE ROLE auditoria
LOGIN
PASSWORD 'Auditoria123#'
NOSUPERUSER
NOCREATEDB
NOCREATEROLE
NOREPLICATION;

4) Permisos necesarios

GRANT CONNECT ON DATABASE postgres TO auditoria;

GRANT USAGE ON SCHEMA auditoria TO auditoria;

GRANT INSERT, SELECT ON auditoria.failed_logins TO auditoria;

GRANT USAGE, SELECT ON SEQUENCE auditoria.failed_logins_id_seq TO auditoria;

5) Configurar autenticación en pg_hba.conf

Editar el archivo:

sudo nano /var/lib/pgsql/17/data/pg_hba.conf

Agregar (recomendado SCRAM):

host    postgres    auditoria    127.0.0.1/32    scram-sha-256
host    postgres    auditoria    ::1/128         scram-sha-256

Reiniciar el servicio:

sudo systemctl restart postgresql-17

Probar conexión manual:

PGPASSWORD='Auditoria123#' psql -U auditoria -d postgres -h localhost -p 5432 -c "SELECT now();"

6) Script para cargar logins fallidos

Crear el archivo:

nano load_failed_logins.sh

Contenido del script:

#!/bin/bash

LOG_DIR="/var/lib/pgsql/17/data/log"

export PGPASSWORD='Auditoria123#'

for logfile in $LOG_DIR/postgresql-*.log
do

grep "password authentication failed" "$logfile" | while read line
do

EVENT_TIME=$(echo "$line" | awk '{print $1" "$2}')

USERNAME=$(echo "$line" | sed -n 's/.*user "\([^"]*\)".*/\1/p')

IP=$(echo "$line" | grep -oE '[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+')

psql -U auditoria -d postgres -h localhost -p 5432 <<EOF

INSERT INTO auditoria.failed_logins
(event_time, username, client_ip, error_message, log_line)
VALUES
(
'$EVENT_TIME',
'$USERNAME',
'$IP',
'password authentication failed',
\$\$$line\$\$
)
ON CONFLICT (log_line) DO NOTHING;

EOF

done

done

Dar permisos de ejecución:

chmod +x load_failed_logins.sh

7) Ejecución recomendada (por permisos de lectura de logs)

Los logs suelen pertenecer a postgres. Para evitar problemas de permisos de lectura, ejecuta: 

sudo -u postgres ./load_failed_logins.sh

8) Validación en la tabla

SELECT
id,
event_time,
username,
client_ip,
inserted_at
FROM auditoria.failed_logins
ORDER BY inserted_at DESC;
